Voice Emojis

Role UX Designer
Team Zheru
Type School Group Project

This projects aims at improving audio message communciation efficency allowing users to input emojis by voice input with our voice emoji keyboard feature. Such design have the potenial to explore the possibility of auditory design in a limited time for Sonification Lab at Georgia Tech.

THE CHALLENGE

Low Efficency in Audio Message Processing

When users use current audio message input, assistive emotion expression tools like emoji are hard to use. Users need to stop audio recording process and switch to a traditional keyboard input method. They need to stop recording message, tap on the emoji selection panel where everything is small, recognize the emoji icons and select the right one. This task is extremely difficult for the ones who don’t have both of their hand free or on a moving action. We aim to design an audio interface to help them use the assistive emotion expression tool like emoji on the audio messaging apps.

Quick Solution

Audio Buffer or Polisher Activated by Voice Input

We aim to use the auditory user interface that involves speech-based input by the user to generate non-speech output (emoji) from the system. The design requirements we generated from persona and previous research include
1) The design will be a seamless user experience;
2) The design will be intuitive enough and easy to learn;
3) The design will be interesting enough and add joy to the texting process.


Use Case 1

Audio message senders need audio buffer or polisher to more accurately convey their emotions.

  • Open the app;
  • Search for Piedmont Park
  • View Piedmont Park Recommended Spots;
  • Use the map to estimate the nearest park spot near Fountain
  • Save
  • Navigate

Use Case 2

Audio message receiver want to quickly estimate the content of received audio messages.

  • Open the app;
  • Search for Fox Theater
  • Search for parking nearest to Fox Theater;
  • Calculate the time needed from navigating to walking back to the theater
  • Navigate

VOI Key Elements Specification

Wake Up Word

With these scenarios in mind, we brainstormed several wake-up mechanisms as follows. Through discussion, we discovered that

  • Concept 1 high accuracy demands in recognizing the Wake-up word. When an recogniztion failure happens, the wake up word will be recorded as part of the audio message which will make error fixation hard for users.
  • Concept 3 is more intuitive than concept 2 since it minimizes user finger movements and won't interrupt natural user audio message input process. Users can switch between audio message record process and emoji recognize process more easily.



Utterance Word

Aside from adding an emoji to an audio message, users also need to delete an emoji, replace an emoji or the system can predict an emoji according to iOS emoji usage guidelines. We identified the following utterances to suit for these typical usage scenarios.

1) Add an emoji. (Default mode)

2) Repeat an emoji.

"Repeat" or " Repeat (n) times." （"Repeat" stands for add the same one twice.）

"Repeat (n) times" （"n" can be customized by users.）

2) Replace an emoji.

" Not this one." /"Wrong."/"No" （Delete the recognized emoji.）

"Next one." （Show the next alternative options.）

"Change." （ Confirm change to user selected one.）

3) Delete an emoji

"Delete."

Emoji Activation Word

To cater for the needs of the largest population to input various audio messages, we considered three alternative voice input methods to ensure all the users can activate the right emoji with their language habits. * Machine learning technology can be used to meet this goal.

1) Names of emoji, eg. "Happy Face."

2) Emotions related to the emoji, eg. "I am super happy now."

3) Visual Features of the emoji, eg. "Emoji with open mouth."

4) Sound impressions of the emoji. eg. "Hahaha."



Interface Design

Main Flow

In the process of wireframe, we discovered that the system needs to put users in control when an emoji is recognized. An emoji wait status above the record button is added, providing users ability to judge and proceed with either confirm, change or repeat. When the emoji is added, it will jump to the message box waiting to be sent out.

Reflections

What I learned from the Projects

Let users talk. At the very begining, we were trapped with our own experience and had a hard time identifying a valueable key points. However, when we went out and talked with the real users, their stories inspired us a lot. I really appreciate that when we were doing the pilot interview, Christa asked the participants, "Besides eating, is there other scenarios that you were experiencing?" And when the visually impaired participants tell her movie stories, we were so excited and inspired. Talk with users allow us to break the border of designers' own life and gain empathy towards our users.

Considering the context when designing tasks. This App serves for a whole process from purchasing movie tickets to walk out the theater. Different contexts have high influence on how each stakeholder behavea. For example, the manager at the Regal Cinema told us, they wouldn't provide service prior the arrival of the users. However, the visually impaired users would love to be prepared before their arrival. Considering different context limitations throughout the process is such a great lesson for me.
